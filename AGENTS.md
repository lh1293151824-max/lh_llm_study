AGENTS.md
本文件用于约定 Codex 在本项目中的执行规则。除非用户明确修改本文件，否则 Codex 必须严格遵守以下要求。
1. 操作范围
	1	当前打开的项目根目录是本次允许操作的范围。
	2	Codex 只能读取、创建、修改、移动、重命名或删除项目根目录以内的文件。
	3	如需访问或操作项目根目录以外的文件，必须提前说明：
	◦	具体路径；
	◦	操作原因；
	◦	计划执行的操作。
	4	只有获得用户明确确认后，才允许访问或操作项目根目录以外的文件。
2. 任务确认
执行任何可能改变项目状态的操作前，Codex 必须先列出勾选式任务清单，并等待用户逐项确认。
需要确认的操作包括但不限于：
	1	创建、修改、删除、移动或重命名文件。
	2	运行可能生成、覆盖、移动或删除文件的代码或命令。
	3	运行会产生缓存、日志、模型、结果或中间文件的程序。
	4	修改配置文件、环境文件、脚本文件或依赖文件。
	5	安装、升级或删除依赖。
	6	执行会改变 Git 状态的操作。
	7	创建、修改或删除临时任务文件。
确认清单格式：
请确认本次任务：

- [ ] 第 1 项：创建并编写临时任务文件 `codex_linshi.py`。
- [ ] 第 2 项：修改 `文件名`，完成具体任务。
- [ ] 第 3 项：运行指定代码，并生成预期结果。
执行规则：
	1	只有用户明确回复“确认执行第 X 项”后，Codex 才能执行对应任务。
	2	未确认的任务不得执行。
	3	用户确认某一项任务，不代表确认其他任务。
	4	相互独立的任务必须分别列出，不得合并成含糊的确认项。
	5	执行过程中出现新增任务或方案变化时，必须暂停并重新获得确认。
	6	不得以“顺便优化”“顺便清理”或“顺便修复”为由扩大任务范围。
3. 临时任务文件
3.1 使用流程
	1	用户确认后，Codex 必须先在项目根目录创建临时任务文件。
	2	临时文件创建完成前，不得修改其他项目文件，也不得运行会改变项目状态的命令。
	3	临时文件必须真实记录本次任务的修改计划，不得在任务完成后补写。
	4	临时文件编写完成后，必须将其内容交给用户检查。
	5	只有用户确认临时文件内容后，才能执行具体任务。
	6	执行方案发生变化时，必须先更新临时文件并重新获得用户确认。
	7	任务完成后，必须询问用户是否删除临时文件。
	8	未获得用户确认前，不得删除临时文件。
3.2 文件名称
默认文件名称：
codex_linshi.py
如果项目中已经存在同名文件：
	1	不得覆盖原文件。
	2	必须向用户说明文件冲突。
	3	等待用户确认新的临时文件名称后再创建。
3.3 文件内容
临时文件使用 Python 注释或纯文本记录任务计划，不要求能够直接运行。
推荐格式：
# ==================== 本次任务 ====================
# 任务名称：
# 用户已确认的任务：

# ==================== config.py 修改伪代码 ====================
# [修改目的：集中管理超参数]
# 1. 新增 learning_rate 配置项。
# 2. 新增 batch_size 配置项。
# 3. 为关键配置添加注释。

# ==================== train.py 修改伪代码 ====================
# [修改目的：从配置文件读取训练参数]
# 1. 导入配置模块。
# 2. 移除主程序中的重要超参数硬编码。
# 3. 增加训练进度显示。

# ==================== 需要讨论与不确定的地方 ====================
# 1. 需要用户确认的事项：
# 2. 当前存在的不确定内容：
# 3. 可能影响任务执行的条件：
# 4. 需要用户选择的实现方案：
临时文件至少应包含：
	1	本次任务目标。
	2	用户已经确认的任务。
	3	预计创建、修改、删除、移动或重命名的文件。
	4	每个文件的修改目的和修改伪代码。
	5	是否需要安装依赖。
	6	是否需要访问项目外文件。
	7	需要用户讨论或确认的事项。
如果没有需要讨论的内容，也必须写明：
# ==================== 需要讨论与不确定的地方 ====================
# 暂无需要讨论或确认的事项。
3.4 伪代码要求
	1	每个待修改文件必须单独列出。
	2	伪代码应说明修改什么、为什么修改以及大致如何实现，伪代码不需要注释
	3	重要函数、类、配置项和文件路径应明确写出。
	4	不得加入未经用户确认的额外功能、重构、格式化或清理任务。
	5	涉及依赖安装时，必须写明：
	◦	依赖名称；
	◦	安装原因；
	◦	安装命令；
	◦	是否修改依赖配置文件；
	◦	是否影响当前环境。
	6	涉及 Git 操作时，必须逐条写明预计执行的 Git 命令。
	7	其他需要确认或讨论的事项，必须统一写在临时文件最后的以下部分：
==================== 需要讨论与不确定的地方 ====================
4. 依赖安装
安装、升级或删除任何依赖前，必须单独获得用户确认。
确认内容必须包括：
请确认是否安装依赖：

- [ ] 依赖名称：
- [ ] 安装原因：
- [ ] 安装命令：
- [ ] 是否修改依赖配置文件：
- [ ] 是否影响当前运行环境：
未经确认，不得：
	1	安装、升级或删除依赖。
	2	执行全局安装或批量升级。
	3	修改 requirements.txt、pyproject.toml、environment.yml、package.json 或其他依赖配置文件。
5. 代码运行
	1	运行任何代码前，必须获得用户确认。
	2	运行代码必须使用 Start-Process 弹出可见的 PowerShell 窗口。
	3	不得使用隐藏窗口或静默后台运行。
	4	所有运行任务必须提供进度条或定期日志。
	5	运行命令、参数、工作目录和预期输出必须提前写入临时任务文件。
	6	运行可能产生文件时，必须提前说明：
	◦	文件名称；
	◦	文件路径；
	◦	文件用途；
	◦	是否可能覆盖已有文件。
	7	运行失败后，应先汇报错误。未经确认，不得连续执行可能改变项目状态的修复命令。
	8	不得擅自终止、重启或重复运行程序。
推荐启动方式：
Start-Process powershell `
    -WorkingDirectory "当前项目根目录" `
    -ArgumentList "-NoExit", "-Command", "python main.py"
推荐进度显示：
from tqdm import tqdm

for i in tqdm(range(total_steps), desc="Running"):
    ...
或：
print(f"当前进度：{current_step}/{total_steps}", flush=True)
6. Git 操作
任何会改变 Git 状态的操作，都必须提前获得用户确认，包括但不限于：
git add
git commit
git push
git pull
git merge
git rebase
git reset
git checkout
git switch
git clean
git stash
创建、删除或修改分支、标签、提交历史和远程仓库，也必须单独确认。
只读 Git 命令可以用于查看状态；如果无法确定命令是否完全只读，则必须先获得用户确认。
7. 禁止事项
Codex 不得：
	1	未经确认修改项目文件或执行会改变项目状态的命令。
	2	擅自访问或操作项目根目录以外的文件。
	3	擅自安装、升级或删除依赖。
	4	擅自运行训练、仿真、推理、批处理或数据转换任务。
	5	在无可见窗口、无日志或无进度反馈的情况下运行代码。
	6	执行临时任务文件中未记录或未经用户确认的修改。
	7	覆盖已有的临时文件或其他项目文件。
	8	擅自删除缓存、日志、模型、结果或中间文件。
	9	进行与当前任务无关的重构、清理或格式化。
	10	隐瞒执行失败、部分完成、未运行测试或结果未经验证的情况。
	11	声称已经完成实际未执行的修改、运行或验证。
	12	将用户对某项任务的确认扩大解释为对其他任务的确认。
8. 任务完成汇报
任务完成后，Codex 必须使用以下格式汇报：
本次任务完成情况：

1. 已完成内容：
2. 修改或生成的文件：
3. 删除、移动或重命名的文件：
4. 执行的命令：
5. 是否运行代码：
6. 代码或测试运行结果：
7. 是否安装、升级或删除依赖：
8. 是否访问项目根目录以外的文件：
9. 临时任务文件当前状态：
10. 实际修改是否与临时文件一致：
11. 尚未完成的内容：
12. 需要用户进一步检查或确认的事项：
根据实际情况补充以下说明：
本次未修改项目文件。
本次未运行任何代码或命令。
本次未安装、升级或删除任何依赖。
本次未访问、读取、修改或删除项目根目录以外的文件。
任务完成后，必须询问：
本次任务已经完成，是否删除临时任务文件 `codex_linshi.py`？
只有用户明确确认后，才能删除临时文件。
9. 指令阅读确认
Codex 完整阅读并理解本文件后，在处理本项目任务时，回复的第一句话必须是：
我完整阅读了AGENTS.md
该句话必须位于回复最开始，其前面不得添加标题、问候语或其他内容。
该句话只表示 Codex 已阅读并理解规则，不表示用户已经确认任何任务。
